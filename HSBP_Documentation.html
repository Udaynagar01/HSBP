Technical Documentation

Detailed system-level explanation of architecture, implementation, and working flow.

1) Project Overview

  • HSBP is a MERN-based web application for booking local home services.
  • Roles supported: Customer, Provider, Admin.
  • End-to-end flow includes service listing, booking lifecycle, approvals, and account management.

2) Technology Stack

Frontend
React, Vite, React Router, Axios
Backend
Node.js, Express.js
Database
MongoDB + Mongoose
Security
JWT + bcryptjs + role middleware

3) High-Level Architecture

  1. React frontend sends API requests to Express backend.
  2. Backend validates JWT and role permissions.
  3. Controllers execute business logic.
  4. Mongoose models read/write MongoDB data.

4) Folder Structure

HSBP/ backend/ src/ config/ # DB connection controllers/ # business logic middleware/ # auth + roles models/ # User, Service, Booking routes/ # API endpoints server.js seed.js frontend/ src/ api/ components/ context/ pages/ theme.js

5) Backend Core Modules

  • User model: name, email, password, role, isApproved, resetToken, resetTokenExpiry.
  • Service model: title, category, description, price, provider reference.
  • Booking model: user, provider, service, address, schedule, status.
  • Auth middleware: `protect` + `allowRoles` for access control.
  • Controllers: auth, service, booking, admin domain handlers.

6) Frontend Core Modules

  • App routing: public + protected routes using `ProtectedRoute`.
  • Layout: role-based sidebar shell for authenticated users.
  • AuthContext: stores current user and login/logout state.
  • Toast system: centralized success/error/info/warning notifications.
  • Axios interceptor: attaches JWT from local storage to API calls.

7) Key API Groups

  • /api/auth: register, login, profile, forgot/reset password
  • /api/services: list/create/my-services/delete
  • /api/bookings: create/my/provider/status-update
  • /api/admin: users/providers approval/bookings/password reset

8) Role-wise Flow

Customer
Login -> browse services -> book -> track/cancel
Provider
Register -> approval -> add service -> manage assigned bookings
Admin
Approve providers -> monitor users/bookings -> reset user passwords

9) Security & Validation

  • Passwords are hashed with bcrypt before storage.
  • JWT tokens are verified for protected endpoints.
  • Role and ownership checks prevent unauthorized updates.
  • Provider self-booking is blocked.
  • Password reset token has expiration.

10) Setup to Run (Local)

# 1) Start MongoDB mongod --dbpath "%USERPROFILE%\HSBP\data" # 2) Backend cd backend copy .env.example .env npm install npm run dev # 3) Seed Admin node src/seed.js # 4) Frontend cd ../frontend npm install npm run dev
Frontend: http://localhost:5173 | Backend: http://localhost:5000

11) Features Implemented

  • Modern home page and improved UI theme.
  • Service search and category filters.
  • Provider booking actions: accept, decline, complete.
  • User cancellation of pending bookings.
  • Provider service delete support.
  • Profile edit page and toast notifications.
  • Forgot/reset password and admin password change.
  • Responsive dashboard layout and role-based navigation.

Non-Technical Documentation

Simple business-level explanation for presentation and viva.

1) What is HSBP?

HSBP is an online platform where people can quickly find and book nearby home services like plumbing, electrical work, cleaning, carpentry, painting, and AC repair.

2) Why this project was built

  • Local service discovery is mostly unorganized and trust is low.
  • Customers often waste time finding reliable providers.
  • HSBP provides one organized system to discover, book, and track services.

3) User Roles and Benefits

Customer
  • Search and book services
  • Track booking status
  • Cancel pending requests
Provider
  • List offered services
  • Receive booking requests
  • Accept/reject/complete jobs
Admin
  • Approve providers
  • Monitor users and bookings
  • Maintain platform quality

4) End-to-End User Journey (Simple)

  1. Customer signs up and logs in.
  2. Customer selects service and places booking.
  3. Provider receives booking and responds.
  4. Job is completed and status is updated.
  5. Admin can supervise and handle issues.

5) Phase-wise Build Journey

  • Phase 1: Setup + login + role system
  • Phase 2: Services + bookings + dashboards
  • Phase 3: UI polish + home page + responsive layout
  • Phase 4: Password reset + profile + admin controls
  • Phase 5: Error fixes + cleanup + demo readiness

6) Value Delivered

  • Improves convenience for customers.
  • Creates digital visibility for local providers.
  • Introduces trust via provider approval workflow.
  • Makes service lifecycle traceable from booking to completion.

7) Current Limitations

  • No payment gateway integration yet.
  • No email provider integration for reset links (demo mode).
  • No rating/review module yet.
  • No production deployment pipeline included.

8) Future Scope

  • Online payments and invoices
  • Email/SMS and real-time notifications
  • Ratings and customer reviews
  • Analytics dashboard and reports
  • Mobile app version
